# Walking Simulator — Feature Plan

Goal: a first-person walking sim minigame (Chants of Sennaar-style word learning in a 3D space),
launched as a desktop app window like everything else. **No 3D physics at all** — the player
is a point that walks on a navmesh authored or generated in Blender, and object picking is
done with an ID render buffer instead of raycasts. There is no 3D collision math anywhere
in the feature.

## What we already have (reuse)

| Existing piece | What it gives the walking sim |
|---|---|
| `UI3DScene` (ld59/UI) | The whole viewport: render target inside a desktop window, pointer-lock mouse look (yaw/pitch), WASD fly movement, crosshair, shadow pass driving. ~80% of the FP camera is already here. |
| Quartz `Scene` + XML loading | Level format. `Scene.FromFile` + `EntityFactory` already load entities with `Position3D` and typed component properties. |
| `Mesh3DComponent` / `BoxPrimitive3DComponent` | Level geometry. FBX-via-content-pipeline is proven (models/ is full of Blender exports). |
| Point/Directional lights + shadow maps | Scene lighting. Realtime cube shadows (4 point lights) + directional ortho shadow. Note: there is **no lightmapping** in the engine today — see Rendering below. |
| Quartz `Physics` (2D) | Barely needed — navmesh walking does no collision resolution. Boundary-edge sliding mirrors the wall-slide feel of the 2D solver, but the code is new (and small). |
| `Scene3DViewerUI` + `Scene3DAsset` | App-shell pattern: GameFile → asset wrapper → scene. `Scene3DAsset` explicitly exists to hold metadata, so a `Mode=Walk` flag slots right in. |
| `AudioAtlas`, `SettingsUI`, dev console | Footsteps, sensitivity option, debug toggles. |

## Key decision: movement model — a point walking on a navmesh

**Requirements: freely modeled terrain (slopes, caves) and no 3D physics.**

The player is a **point constrained to a navmesh surface**. The navmesh — hand-modeled in
Blender or generated by a Recast-style addon — already encodes everything a physics system
would otherwise compute:

- **Boundary edges are the walls.** Moving across an interior (shared) edge enters the
  neighboring triangle; hitting a boundary edge slides the remaining movement along it.
  There is no collision detection or resolution — movement is a walk across the triangle
  adjacency graph.
- **Triangle planes are the floor.** `Y` is evaluated from the current triangle's plane each
  frame, so slopes and uneven terrain come from the mesh itself. No gravity, no grounded
  check, no max-slope policy — if it's in the navmesh, it's walkable, by construction.
- **Caves/overlaps are unambiguous** because the controller tracks its current triangle and
  only ever moves to adjacent ones; two surfaces overlapping in XZ (tunnel under a hill)
  never interact.
- **Player radius is an authoring-time concern:** inset the navmesh from walls by ~0.35 m
  when modeling it (Recast-style generators do this automatically from an agent-radius
  parameter). The runtime stays a point.
- The player physically cannot fall through geometry, clip through walls, or leave the
  playable area — there is no mechanism by which any of that could happen.

Earlier candidates, for the record: Obra Dinn-style flat zones + ramp connectors (rejected:
hand-authored per level, can't do sculpted slopes) and kinematic ellipsoid vs level triangles
(rejected: it's writing a small 3D physics engine — penetration resolve, gravity, slope
policy — that a navmesh makes unnecessary). The navmesh is the synthesis: generated/authored
data, trivial runtime.

## New features to build

### 1. Navmesh runtime
- Asset: the navmesh exports from Blender as `.obj` (a ~20-line parser; no content-pipeline
  involvement). Weld vertices within an epsilon on load, then build triangle adjacency from
  shared vertex pairs. `Scene3DAsset` gains a `NavMeshPath` element.
- `NavMesh` class:
  - `FindTriangle(Vector3 pos)` — linear scan, used only at spawn.
  - `Move(int tri, Vector3 pos, Vector2 deltaXZ)` → new position + triangle. The edge-walk:
    advance in XZ; if the segment exits the current triangle through a shared edge, continue
    into the neighbor with the remaining delta; if through a boundary edge, clip to the edge
    and slide the remainder along it. Iteration cap + epsilon nudge at vertices (the one
    classic fiddly spot — handle exiting exactly through a corner).
  - Height: barycentric/plane evaluation of `Y` on the current triangle.
- All pure functions (point-in-triangle in XZ, segment-edge crossing, edge slide, plane
  height) — fully unit-testable without rendering. No spatial index needed, ever: movement
  only touches the current triangle and its neighbors.
- No 3D collision math at all: object picking is the ID-buffer pass (section 4), and trigger
  volumes are a point-in-box check of the player position against `BoxCollider3DComponent`
  (Size, Offset, IsTrigger) boxes.

### 2. Character controller
- `WalkController`: current triangle index + position on mesh, eye height, move speed,
  acceleration.
- Per frame:
  1. Input → desired XZ delta along camera-flattened forward/right.
  2. `NavMesh.Move(...)` — that's the entire "physics".
  3. Camera `Y` = triangle-plane height + eye height, lightly smoothed (SmoothDamp) so
     stair-shaped navmesh ribbons read as steps instead of pops.
- No gravity, no jumping, no falling, no grounded state — none of it exists to go wrong.

### 3. First-person camera (refactor `UI3DScene`)
- Extract the mouse-look/pointer-lock code into a shared piece, or add
  `CameraMode { Fly, Walk }` to `UI3DScene`.
  - Fly = current behavior (keep for the scene viewer app).
  - Walk = WASD drives the controller on XZ (forward flattened to the horizontal plane),
    persistent capture (click to grab, Esc to release — already mostly how it works).
- Feel (later): head bob, sprint, FOV setting, sensitivity in `SettingsUI`.

### 4. Interaction system — ID-buffer picking (no raycasts)
Hover detection is a render pass, not geometry math — reusing what the engine already does
well:
- Render the scene to a small offscreen **ID buffer** with a flat unlit color per entity:
  each interactable gets its assigned ID encoded as a color (ID → red channel; 255 IDs is
  plenty), everything else renders black (ID 0). Rendering *all* geometry — not just
  interactables — makes occlusion correct for free via the depth buffer: you can't hover a
  sign through a wall.
- This is the existing replacement-effect pattern: `Scene.DrawShadowPass` +
  `Mesh3DComponent.DrawDepth` already render every entity with a substitute shader. A tiny
  `id-color.fx` (flat color out, same vertex transform) is the only new shader.
- Sample the pixel under the crosshair with `RenderTarget2D.GetData`. Keep the RT small
  (e.g. 160×90 — we only ever read the center pixel) and/or sample every 2–3 frames:
  CPU readback forces a GPU sync, and a tiny RT at hover cadence makes that free in practice.
- Range gate: a plain distance check from player to the hovered entity's `Position3D`
  (~2.5 m). No ray needed.
- Perks over ray-vs-box picking: it's pixel-perfect (follows the actual mesh silhouette —
  exactly right for pointing at individual glyphs/signs), and the hovered ID makes highlight
  rendering trivial (tint that entity in the main pass).
- `Interactable3DComponent`: ID assigned at scene load (ID → entity map), prompt text,
  interact action.
- Actions bridge back into the desktop game — this is the point of the whole feature:
  - show a glyph/word (vocabulary learning),
  - unlock/open a `GameFile`,
  - fire a `NotificationPopup` / story flag.
- Trigger volumes for walk-into events: player point-in-box vs `BoxCollider3D` IsTrigger
  boxes.

### 5. Level authoring

#### Design a whole level in Blender (recommended)
`tools/blender_export_walksim.py` exports the visible Blender scene as a ready-to-walk level in
one run: mesh objects → FBX + `Mesh3D`, marked objects → `Interactable3D`, walkable meshes →
a navmesh (baked via `NavMeshBaker`), point/sun lights, a spawn, plus the scene XML +
`Scene3DAsset` (`Mode=Walk`) + `.scene3d`, and patches `Content.mgcb`. Meter-scaled
(`SceneScale=1`). Set `GAME_CONTENT_DIR`/`REPO_DIR`/`OUTPUT_SCENE_NAME` at the top, run in
Blender's Scripting workspace, then rebuild the game.

Mark objects with **custom properties** (Object Properties → Custom Properties):

| Property | On | Effect |
|---|---|---|
| `interact_action` | mesh | marks it interactable; the dispatcher verb (e.g. `show-text`) |
| `interact_prompt` | mesh | crosshair hint (default "interact") |
| `interact_target` | mesh | what the action acts on (file path / glyph id / entity name) |
| `interact_message` | mesh | payload text (for `show-text`, the text shown) |
| `no_collide` = 1 | mesh | exclude from the navmesh (props you can't walk on) |
| `player_start` = 1 | any | spawn point (or just name an object `PlayerStart`) |
| `intensity`, `range`, `shadows` | lights | override defaults (range = point only, shadows = sun only) |

Interaction is data-driven: `Action` flows Blender → `Interactable3DComponent` →
`InteractionDispatcher.Dispatch` (`ld59/WalkingSim/`), which switches on `Action`. Today only
`show-text` is wired; add new effects as one `case` there (the file lists the wireable menu:
reveal/open a file, play a sound, unlock a clue, learn a glyph, toggle an entity).

Because Blender levels are `Mesh3D` throughout, terrain **occludes** ID-buffer picks (no
hover-through-walls) and gets full point + directional lighting and shadows — unlike the raw-OBJ
`ObjTerrain` path. Do the marker-cube check on your first export (below).

#### Powergrid puzzle panels (internals)
`PuzzlePanelComponent` (`ld59/WalkingSim/`) hosts a powergrid puzzle on an object. It loads the
puzzle scene, renders a (view-only) `PowergridView` to a `RenderTarget2D` in an offscreen pass in
`UI3DScene.Update` (before the main pass), and draws a world-mounted textured quad
(`shaders/textured-quad.fx`, unlit) at the entity, oriented by `PanelYaw` (a fixed part of the
level, not a billboard). On `OnInteract`, `WalkingSimUI` opens a
`PuzzleSolveOverlay` — the **same** `PowergridView` resized to a large centred rect with the walk
camera capture suspended (`UI3DScene.SuspendCapture`), so the real cursor solves it. The overlay
polls `Controller.IsLevelSolved` (no events exist), and on the rising edge reads the solution
(`Graphs[].Nodes[].Rune`), matches it against the `Outcomes` DSL (which-solution mapping), and
fires the effect through `InteractionDispatcher`. Environment effects (`reveal`/`hide`/`toggle`)
flip `Entity.Visible`, gated in `Entity.Draw3D` and the pick pass. Note: revealed **walkable**
geometry must be pre-baked into the navmesh (hidden only visually) — the navmesh is static.

#### Hand-authored scene XML
- A level in scene XML is: `Mesh3D` visuals, a navmesh `.obj`, `Interactable3D` (+ optional
  `BoxCollider3D` triggers), lights, and a `PlayerStart` entity.
- Navmesh, two routes:
  - **Generate it with `NavMeshBaker` (default)** — the offline Recast tool (`NavMeshBaker/`,
    built on DotRecast) bakes the walkable navmesh from level geometry. The ~0.35 m wall inset,
    stair/slope walkability, and tunnel headroom all fall out of the agent parameters.
  - **Hand-model it** — duplicate the floor faces, simplify, inset ~0.35 m from walls, cover
    stairs with a ramp ribbon. Only needed for total control; the baker is the normal path.
- One-time sanity check: export a marker cube in both the FBX visuals and the navmesh input and
  confirm they land in the same place in-game — catches axis-convention mismatches on day one.
- Debug (dev-console toggles): draw the navmesh wireframe in-world with the current triangle
  highlighted; noclip = switch the camera back to Fly mode (already exists).

#### NavMeshBaker (Recast) — baking a navmesh
Offline console tool (`NavMeshBaker/`, DotRecast). DotRecast is referenced only here, never in
the shipped game — the game just loads the OBJ the baker writes (via `Scene3DAsset.NavMeshPath`).

```
cd NavMeshBaker
dotnet run -- <input.obj | scene.xml>  <output.obj>  [options]
  --radius <m>   agent radius / wall inset (default 0.35)
  --height <m>   agent height / min headroom (default 1.8)
  --step <m>     max step climb (default 0.3; must exceed stair rise)
  --slope <deg>  max walkable slope (default 45)
  --cell <m>     XZ voxel size (default 0.15); --cellHeight for Y
  --validate     load the result back and report connectivity
```

Two input sources (auto-detected by extension):
- **`.obj`** — a Blender-exported level (or a simplified collision mesh). The natural
  "navmesh from a 3D model" path.
- **scene `.xml`** — bakes from the level's `BoxPrimitive3D` entities (blockouts, no Blender
  step). `Mesh3D` entities are ignored (their geometry is in FBX, not read).

Output is the detail mesh (terrain-following triangles), vertex-welded so seams stay walkable.
The demo `walk_level_navmesh_baked.obj` is produced this way from `walk_level.xml`.

**Getting the model in as visible terrain:** do NOT load it as a MonoGame `Model` — the OBJ/FBX
content pipeline (`OpenAssetImporter`) emits a `MaterialContent` that has no runtime reader, so
`Content.Load<Model>` throws (`Could not find ContentTypeReader ... MaterialContent`) and the
level shows as a blank window. Instead `/copy:` the OBJ into content and point an
`ObjTerrain` component at it (`Property Name="ObjPath"`) — it builds the vertex buffer itself
(flat-shaded) and draws with the `point-light` shader, no materials involved. The same OBJ feeds
the navmesh baker, so visual and navmesh always align. See `terrain.xml` / `terrain_asset.xml`.

Gotchas (all learned the hard way, covered by `WalkingSimTests`):
- **Walkable faces need up-facing normals.** Recast's slope test is winding-sensitive (unlike
  the runtime `NavMesh`). Blender exports up-facing floor normals by default; a floor wound the
  other way bakes to *nothing*. Box-scene input handles this automatically.
- **Overlapping surfaces need ≥ agent-height clearance.** A floor under a platform only stays
  walkable if the gap ≥ `--height` (the demo tunnel is 2.0 m vs 1.8 m — tight but fine). This
  is a design constraint the hand-authored mesh didn't have.
- **Stairs**: `--step` must exceed the step rise or the staircase bakes as a wall.
- **Units/axes**: the baker works in the input's coordinate space; an OBJ must already match the
  scene's world axes/scale (walk scenes use SceneScale = 1, so box-scene input is automatic).

### 6. App shell + integration
- `WalkingSimUI` window modeled on `Scene3DViewerUI`; reuse `FileType.Scene3D` with a
  `Mode` element in `Scene3DAsset` (that wrapper exists precisely for metadata like this).
- Player spawn point = an entity in the scene XML (`Name="PlayerStart"`).
- Footstep audio (distance-interval driven) via `AudioAtlas`; ambient loop per scene.

### 7. Rendering notes
- No lightmapping exists — `UI3DScene.Update` re-renders all shadow cube faces **every frame**.
  Fine for the tiny viewer scenes; for a walking-sim level, cache the shadow maps: render once
  at load, re-render only when a light or entity moves. That's the cheap "baked lighting" win
  and probably the only perf work needed.
- Larger scenes: raise far plane thoughtfully, consider fog or the existing one-bit dithering
  post-process for atmosphere (hides low-poly assets nicely).

## Build order (each step is runnable)

1. Navmesh math as pure functions with unit tests (follow the `PinballSimTests`/`TcgSimTests`
   pattern → `WalkingSimTests`): point-in-triangle, edge-walk move with boundary slide, plane
   height, vertex-corner cases. The whole movement model is testable without rendering.
2. `.obj` loader + vertex weld + adjacency build (also unit-tested — feed it a hand-written
   two-triangle mesh).
3. Walk mode in `UI3DScene` — walk (unconstrained) around `test_3d_scene.xml` at fixed eye
   height.
4. Constrain to navmesh: first a flat square you can't walk off, then the blockout acceptance
   level — sloped terrain, a staircase ribbon, and a cave tunnel passing under walkable
   ground. Proves the whole requirement end to end.
5. Interaction: ID-buffer pass (`id-color.fx` + tiny render target + center-pixel read) +
   prompt + one real action (open a file / show a word).
6. Authoring pass: navmesh wireframe debug view, settle the Blender workflow (hand-model vs
   Recast addon), noclip toggle.
7. Feel: footsteps, head bob, sensitivity setting, ambient audio.
8. Perf: static shadow-map caching (movement needs nothing — it's adjacency walking).

Steps 1–5 are the engine work (~the "walking sim exists" milestone); 6–8 are content and polish.
